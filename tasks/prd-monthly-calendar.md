# PRD: 깔끔한 먼슬리 캘린더 앱 (Monthly Calendar App)

## 1. Introduction / Overview

드래그 기반으로 빠르게 할일을 등록하고, 월간(먼슬리) 캘린더 위에 막대 블럭으로 일정을 한눈에 보는 **깔끔한 개인 일정 관리 앱**이다.

핵심 사용 흐름은 다음과 같다.

1. 2026년 먼슬리 캘린더가 화면에 보인다.
2. 일칸을 드래그하면(여러 날에 걸쳐 가능) 할일 등록 창이 뜬다.
3. 할일 텍스트를 입력하고 저장하면 캘린더에 막대 블럭으로 표시된다.
4. 등록된 할일은 오른쪽 사이드 패널에 날짜순으로 쌓인다.
5. 막대는 클릭(편집)하고 드래그해서 다른 날짜로 옮길 수 있다.
6. 프로젝트별로 각자의 먼슬리 캘린더를 가질 수 있고, 합쳐 보거나 따로 볼 수 있다.

할일은 **두 개의 분류 축**을 가진다 — **프로젝트**(어느 캘린더에 속하나)와 **태스크 종류**(무슨 성격의 일인가). 프로젝트는 **고유 색(같은 톤 8색조 중 하나)**으로, 태스크 종류는 그 색의 **톤 단계(마감=진함 → 리서치=옅음)**로 표현된다. 또한 기간 막대(할일)와 별개로 특정 날짜에 **이벤트/하드 데드라인 마커**를 흑백 기호로 표시한다.

**디자인 언어:** 스위스 에디토리얼 — **모노크롬(흑백) UI + 하이라인**, **선 없는 미니멀 그리드**, **세로 무한 스크롤**(월이 끊김 없이 이어짐). **색은 오직 할일 막대(데이터)에만** 써서 흑백 위에서 도드라지게 한다. 타이포는 영문 Plain 400 / 숫자·한글 IBM Plex Sans KR(무료), 자간 -3%. (시각 레퍼런스: [docs/design/mockups/calendar-layouts.html](../docs/design/mockups/calendar-layouts.html) F 탭)

**해결하는 문제:** 기존 캘린더 앱은 일정 등록 단계가 많고(클릭→폼→여러 필드), 기간 일정과 분류 구분이 번거롭다. 이 앱은 "드래그 한 번 → 텍스트 한 줄"의 최소 마찰로 기간 일정을 등록하고, 색으로 프로젝트와 태스크 종류를 동시에 구분하며, 통합/분리 뷰를 자유롭게 전환하는 것을 목표로 한다.

## 2. Project Context

- **Tech Stack**: Next.js 16 (App Router) + TypeScript + React
- **로컬 저장소**: Dexie (IndexedDB 래퍼) — 서버/로그인 없음, 브라우저 로컬 저장
- **스타일**: Tailwind CSS
- **색상 처리**: HSL 기반 색 계산 유틸(`lib/color/`) — 프로젝트 색조 + 태스크종류 톤 합성
- **드래그 처리**: 포인터 이벤트 기반 직접 구현 (라이브러리 도입 시 dnd-kit 검토)
- **Package Manager**: npm
- **Test Framework**: Vitest + React Testing Library (단위/로직), 브라우저 수동 검증(UI)
- **배포**: Vercel (정적/SSG)
- **대상 기기**: 데스크톱(마우스 드래그) 우선. 모바일 터치는 v1 범위 외(best-effort)
- **폴더 구조(예정)**:
  ```
  app/                # Next.js App Router 페이지
  components/         # 캘린더, 막대, 사이드 패널 등 UI
  lib/db/             # Dexie 스키마 및 데이터 접근 함수
  lib/calendar/       # 날짜 계산, 막대 레이아웃(레인) 로직
  lib/color/          # 색조+톤 합성, 텍스트 대비 계산
  lib/types.ts        # 공용 타입
  ```

## 3. Goals

- 드래그 1회 + 텍스트 1줄로 기간 할일을 등록할 수 있다 (최소 마찰).
- 여러 날에 걸친 할일을 캘린더 위에 연속된 막대 블럭으로 정확히 표시한다.
- 겹치는 할일이 서로 가리지 않게 자동으로 레인(lane)으로 쌓아 보여준다.
- 막대를 드래그해 날짜를 옮기면 기간 길이를 유지한 채 이동한다.
- **하나의 막대 색으로 두 축을 동시에 표현한다**: 색조 = 프로젝트, 명도·채도 = 태스크 종류.
- 통합 뷰에서 색조로 프로젝트가 구분되고, 같은 프로젝트 내에서는 톤으로 태스크 종류가 구분된다.
- 새로고침 후에도 모든 데이터가 로컬에 유지된다.

## 4. 색상 시스템 — 핵심 컨셉

전체 값·HEX·WCAG 매트릭스는 [docs/design/color-system.md](../docs/design/color-system.md)가 정본. 요약:

### 원칙: 모노크롬 셸 + 색은 막대에만

UI(그리드·타이포·사이드바)는 흑백/하이라인. **색은 오직 할일 막대(데이터)에만** 사용 → 흑백 위에서 색이 또렷.

### 프로젝트 = 같은 톤 8색조

모든 프로젝트 색은 **동일 채도·명도(S 58%, L 46%)**, 색조만 다른 8개: 레드 `#B9313D` · 오렌지 `#B97131` · 앰버 `#B99E31` · 그린 `#31B96A` · 틸 `#31B5B9` · 블루 `#3175B9` · 인디고 `#5A31B9` · 퍼플 `#B9319E`. 베이스 명도가 균일해 톤 사다리가 8색 모두 일관.

### 태스크 종류 = 원색의 톤(dark/tint) 단계 (전역 공유)

```
막대 배경색 = applyTone(project.color, taskType)
  dark: out = base × (1 − k)         (마감: k 0.40 — 가장 진함)
  tint: out = base × (1 − k) + 255k  (작업 0.32 / 회의 0.56 / 리서치 0.80)
막대 텍스트색 = 배경 명도 기준 흰/검 자동 (barText)
```

- **검증:** 8색조 × 4톤 = 32조합 전부 WCAG AA(4.5:1) 이상 (자동 흑/백 텍스트).
- 원색(8색)은 프로젝트 정체성 → 범례/탭/점에 그대로. 막대는 톤 적용본.

### 날짜 마커 (이벤트 / 하드 데드라인) — 색이 아닌 흑백 기호

기간 막대와 별개의 점(날짜) 표시. **하드 데드라인** = 검은 채운 칩 `⚑ 라벨`, **이벤트** = 외곽선 칩 `◆ 라벨`. 색상 막대와 구분.

### 범례 (Legend)

- **프로젝트 범례**: 8색 원색 스와치 + 이름 (사이드바).
- **태스크종류 범례**: 한 색의 톤 단계(마감→리서치) + 이름.
- **마커 범례**: ⚑ 데드라인 / ◆ 이벤트.

## 5. User Stories

> 순서는 의존성(스키마 → 데이터 → 캘린더 렌더 → 생성 → 색상 → 막대 → 패널 → 이동 → 프로젝트/태스크종류) 기준이다.

### US-001: 프로젝트 스캐폴딩
**Description:** 개발자로서 Next.js + TypeScript + Tailwind + Dexie 기반 프로젝트 골격을 만들어, 이후 기능을 올릴 토대를 갖춘다.

**Acceptance Criteria:**
- [ ] `npx create-next-app`(App Router, TS, Tailwind)으로 초기화된다.
- [ ] Dexie가 설치되고 `lib/db/`에 빈 DB 인스턴스가 생성된다.
- [ ] `npm run dev`로 빈 홈 화면이 정상 렌더된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 개발 서버에서 페이지가 에러 없이 뜬다.

**Priority:** 1

### US-002: 데이터 모델 & Dexie 스키마
**Description:** 개발자로서 Project/TaskType/Task/Marker 스키마와 데이터 접근 함수를 정의해, 모든 기능이 일관된 로컬 저장소를 사용하게 한다.

**Acceptance Criteria:**
- [ ] `Project { id, name, color(HEX), visible, order, createdAt }` 스키마 정의 (color = 같은 톤 8색조 중 하나)
- [ ] `TaskType { id, name, mode('dark'|'tint'), k(number), order, createdAt }` 스키마 정의 — **전역 공유**
- [ ] `Task { id, projectId, taskTypeId, title, startDate(YYYY-MM-DD), endDate(YYYY-MM-DD), createdAt, updatedAt }` 스키마 정의
- [ ] `Marker { id, kind('event'|'deadline'), label, date(YYYY-MM-DD), projectId?(optional), createdAt }` 스키마 정의
- [ ] 최초 실행 시 기본 프로젝트 1개("기본", 8색 중 하나)와 기본 태스크종류 4개(마감 dark0.40 / 작업 tint0.32 / 회의 tint0.56 / 리서치 tint0.80)가 시드된다 — docs/design/color-system.md 값 사용
- [ ] Project/TaskType/Task/Marker CRUD 함수 제공
- [ ] 데이터가 IndexedDB에 저장되어 새로고침 후에도 유지된다.
- [ ] Typecheck 통과 + CRUD 단위 테스트 통과

**Priority:** 1

### US-003: 무한 스크롤 먼슬리 캘린더 (선 없는 그리드)
**Description:** 사용자로서 월이 세로로 끊김 없이 이어지는 캘린더를 스크롤하며 본다. 페이징(이전/다음) 대신 무한 스크롤.

**Acceptance Criteria:**
- [ ] 7열(일~토) × 주 단위로 여러 달이 **세로로 연속** 렌더된다 (월 그룹은 그 주의 목요일이 속한 월 기준).
- [ ] 각 월 시작에 **sticky 월 헤더**(월 이름 + 요일 행)가 있어 스크롤 시 상단에 고정된다.
- [ ] 로드 시 **오늘(2026-05-21)이 보이도록 자동 스크롤**된다.
- [ ] 상단 타이틀이 현재 스크롤 위치의 월로 갱신되고, "오늘" 버튼으로 오늘로 복귀한다.
- [ ] 위/아래로 스크롤하면 인접 월이 추가로 나타난다(범위 확장/지연 로딩 허용).
- [ ] **선 없는 미니멀 그리드**: 셀 테두리 없음, 큰 날짜 숫자 + 여백으로 구조. 오늘은 검은 원형, 타 월 날짜는 흐리게.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 월 경계가 끊김 없이 이어지고 오늘로 스크롤된다.

**Priority:** 2

### US-004: 일칸 드래그로 기간 선택 → 등록 창 열기
**Description:** 사용자로서 캘린더에서 시작 칸부터 종료 칸까지 드래그하면, 선택한 기간이 미리 채워진 할일 등록 창(팝오버)이 뜬다.

**Acceptance Criteria:**
- [ ] 한 칸을 누른 채 드래그하면 지나간 날짜 칸들이 선택 하이라이트된다.
- [ ] 드래그를 놓으면 시작일~종료일이 정해지고 등록 창이 열린다.
- [ ] 한 칸만 클릭하면 하루짜리(시작=종료) 등록 창이 열린다.
- [ ] 등록 창에 선택된 기간(예: "5/21 ~ 5/24")이 표시된다.
- [ ] 역방향 드래그(종료→시작)도 정상 정규화된다.
- [ ] ESC 또는 바깥 클릭으로 등록 창이 닫힌다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 드래그 범위와 표시 기간이 일치한다.

**Priority:** 2

### US-005: 할일 생성 (프로젝트 + 태스크종류 선택) & 막대 렌더
**Description:** 사용자로서 등록 창에 제목을 입력하고 프로젝트·태스크종류를 고른 뒤 저장하면, 캘린더에 해당 기간을 덮는 막대가 즉시 나타난다.

**Acceptance Criteria:**
- [ ] 제목 입력 필드가 있고, 빈 제목은 저장이 막힌다(검증 메시지).
- [ ] 프로젝트 선택 + 태스크종류 선택 UI가 있다(현재 뷰의 프로젝트가 기본값).
- [ ] 저장 시 막대가 생성·저장되고 시작일~종료일 기간을 가로로 덮는다.
- [ ] 주(week) 경계를 넘는 기간은 각 주 행에 연속 막대로 분할 표시된다.
- [ ] 저장 후 등록 창이 닫히고 막대에 제목 텍스트가 보인다.
- [ ] 새로고침 후에도 막대가 유지된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 시나리오 2~3단계(드래그→입력→막대)가 완결된다.

**Priority:** 3

### US-006: 막대 색상 계산 (원색 + 톤)
**Description:** 개발자/사용자로서 막대 색이 프로젝트 원색에 태스크종류 톤(dark/tint)을 적용해 결정되고, 텍스트가 항상 읽히도록 한다.

**Acceptance Criteria:**
- [ ] 막대 배경색 = `applyTone(project.color, taskType)` (dark: ×(1−k), tint: 흰색 k 혼합)으로 계산된다.
- [ ] 막대 텍스트색이 배경 명도에 따라 자동(흰/검)으로 정해져 가독성(WCAG AA)을 확보한다.
- [ ] 같은 프로젝트의 서로 다른 태스크종류 막대가 톤(진하기)으로 구분된다.
- [ ] 서로 다른 프로젝트의 같은 태스크종류 막대가 색으로 구분된다.
- [ ] 색 합성·대비 계산 로직에 단위 테스트가 있다(color-system.md 32조합 매트릭스와 일치).
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** color-system.md 매트릭스처럼 색이 구분된다.

**Priority:** 3

### US-007: 겹치는 막대 레인(lane) 스택 레이아웃
**Description:** 사용자로서 같은 날짜에 여러 할일이 겹쳐도 막대가 서로 가리지 않고 위아래로 쌓여(레인) 모두 보인다.

**Acceptance Criteria:**
- [ ] 한 주 행에서 기간이 겹치는 막대들은 서로 다른 세로 레인에 배치된다.
- [ ] 겹치지 않으면 같은 레인을 재사용해 공간을 절약한다.
- [ ] 한 칸에 막대가 너무 많으면 "+N개" 형태로 요약하고 클릭 시 펼친다.
- [ ] 레이아웃 계산 로직에 단위 테스트가 있다(겹침/비겹침/경계 케이스).
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 겹치는 일정 3개 이상이 모두 보인다.

**Priority:** 3

### US-008: 오른쪽 사이드 패널 — 날짜순 할일 목록
**Description:** 사용자로서 화면 오른쪽 패널에서 등록된 할일을 날짜순으로 쌓인 목록으로 본다.

**Acceptance Criteria:**
- [ ] 우측 고정 패널에 할일이 시작일 오름차순으로 정렬되어 나열된다.
- [ ] 각 항목에 제목, 기간, 프로젝트(색조)·태스크종류(톤/라벨)가 표시된다.
- [ ] 현재 보이는 프로젝트/뷰 필터와 목록이 동기화된다.
- [ ] 패널의 항목을 클릭하면 캘린더의 해당 막대가 강조(스크롤/하이라이트)된다.
- [ ] 할일이 없으면 빈 상태 안내가 표시된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 할일 추가 시 패널이 즉시 갱신된다.

**Priority:** 4

### US-009: 막대 클릭 → 편집 / 삭제
**Description:** 사용자로서 막대를 클릭해 제목·기간·프로젝트·태스크종류를 수정하거나 할일을 삭제할 수 있다.

**Acceptance Criteria:**
- [ ] 막대 클릭 시 편집 창이 열리고 기존 값이 채워져 있다.
- [ ] 제목/시작일/종료일/프로젝트/태스크종류 변경이 저장된다.
- [ ] 프로젝트/태스크종류 변경 시 막대 색(색조/톤)이 즉시 반영된다.
- [ ] 삭제 버튼은 확인 단계(되돌릴 수 없음 안내) 후 삭제한다.
- [ ] 편집/삭제 결과가 캘린더와 사이드 패널에 즉시 반영된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 편집·삭제가 양쪽 뷰에 반영된다.

**Priority:** 4

### US-010: 막대 드래그로 날짜 이동
**Description:** 사용자로서 막대를 잡고 다른 날짜로 드래그하면, 기간 길이를 유지한 채 일정이 이동한다.

**Acceptance Criteria:**
- [ ] 막대를 드래그하는 동안 이동 위치가 미리보기로 표시된다.
- [ ] 놓으면 시작일이 옮겨지고 종료일은 동일한 기간(일수)을 유지한다.
- [ ] 주 경계를 넘는 이동도 정확히 재배치된다.
- [ ] 이동 결과가 DB에 저장되고 새로고침 후 유지된다.
- [ ] (편집 클릭과 구분) 짧은 클릭은 편집, 드래그는 이동으로 동작한다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 시나리오 5단계(드래그 이동)가 동작한다.

**Priority:** 5

### US-011: 프로젝트 관리 (생성 / 이름변경 / 색 / 삭제)
**Description:** 사용자로서 프로젝트를 만들고 이름과 색(같은 톤 8색조 중 선택)을 지정하거나 삭제할 수 있다.

**Acceptance Criteria:**
- [ ] 프로젝트 추가 시 이름 입력 + 색 선택(8색 팔레트)이 가능하다.
- [ ] 미사용 색을 기본 추천값으로 제시한다.
- [ ] 프로젝트 이름/색 변경이 저장되고 관련 막대 색에 즉시 반영된다.
- [ ] 프로젝트 삭제 시 소속 할일 처리(함께 삭제 또는 기본 프로젝트로 이동)를 확인 단계로 묻는다.
- [ ] 기본 프로젝트는 삭제할 수 없다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 색 변경이 막대에 반영된다.

**Priority:** 5

### US-012: 태스크 종류 관리 (전역 / 이름 / 톤 / 삭제)
**Description:** 사용자로서 전역 태스크 종류를 만들고 이름과 톤(명도·채도 단계)을 지정하거나 삭제할 수 있다.

**Acceptance Criteria:**
- [ ] 태스크종류 추가 시 이름 입력 + 톤 단계 선택(진함↔연함)이 가능하다.
- [ ] 태스크종류는 모든 프로젝트에서 공유된다.
- [ ] 이름/톤 변경이 저장되고 관련 막대에 즉시 반영된다.
- [ ] 태스크종류 삭제 시 소속 할일 처리(기본 종류로 이동)를 확인 단계로 묻는다.
- [ ] 기본 태스크종류는 삭제할 수 없다.
- [ ] (안내) 권장 단계 수(약 3~4개)를 넘으면 구분이 어려워질 수 있음을 알린다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 톤 변경이 막대에 반영된다.

**Priority:** 5

### US-013: 뷰 전환 (개별 프로젝트 ↔ 통합) + 색 구분
**Description:** 사용자로서 특정 프로젝트만 보거나 모든 프로젝트를 합친 통합 캘린더를 볼 수 있고, 통합 뷰에서 프로젝트는 색으로 구분된다. 상단 프로젝트 탭(전체 + 프로젝트들, 밑줄형) IA.

**Acceptance Criteria:**
- [ ] 상단 프로젝트 탭에서 "전체(통합)" 또는 개별 프로젝트를 선택할 수 있다.
- [ ] 통합 뷰는 모든 프로젝트의 막대를 색으로 구분해 함께 표시한다.
- [ ] 개별 뷰는 해당 프로젝트의 막대만 표시한다(태스크종류 톤은 유지).
- [ ] 새 할일 등록 시 현재 선택된 프로젝트가 기본값이 된다(통합 뷰에선 선택 필요).
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 뷰 전환 시 막대 집합이 올바르게 바뀐다.

**Priority:** 6

### US-014: 사이드바 범례 + 프로젝트 보이기/숨기기 토글
**Description:** 사용자로서 사이드바에서 두 축의 범례를 보고, 통합 뷰에서 프로젝트별 토글로 표시/숨김을 제어한다.

**Acceptance Criteria:**
- [ ] 사이드바(또는 우측 할일 리스트)에 프로젝트 범례(8색 스와치 + 이름 + 켜기/끄기 토글)가 나열된다.
- [ ] 태스크종류 범례(톤 단계 + 이름)와 마커 범례(⚑ 데드라인 / ◆ 이벤트)가 표시된다.
- [ ] 프로젝트 토글을 끄면 해당 프로젝트 막대가 캘린더·사이드 패널에서 숨겨진다.
- [ ] 토글 상태(visible)가 저장되어 새로고침 후에도 유지된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 토글 on/off가 즉시 반영된다.

**Priority:** 6

### US-015 (선택): 태스크 종류 필터
**Description:** 사용자로서 태스크종류 단위로도 막대를 필터링(특정 종류만 보기)할 수 있다.

**Acceptance Criteria:**
- [ ] 태스크종류 범례에서 종류별 켜기/끄기로 막대를 필터링한다.
- [ ] 프로젝트 필터와 조합(AND)으로 동작한다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 종류 필터가 즉시 반영된다.

**Priority:** 7

### US-016 (선택): 막대 가장자리 드래그로 기간 조절
**Description:** 사용자로서 막대의 좌/우 끝을 잡고 드래그해 시작일·종료일을 늘리거나 줄일 수 있다.

**Acceptance Criteria:**
- [ ] 막대 좌측 끝 드래그 시 시작일이, 우측 끝 드래그 시 종료일이 변경된다.
- [ ] 시작일이 종료일을 넘지 못한다.
- [ ] 변경이 저장되고 즉시 반영된다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 리사이즈가 동작한다.

**Priority:** 7

### US-017: 날짜 마커 (이벤트 / 하드 데드라인)
**Description:** 사용자로서 특정 날짜에 이벤트 또는 하드 데드라인 마커를 추가하고, 캘린더에 흑백 기호로 표시한다(기간 막대와 별개).

**Acceptance Criteria:**
- [ ] 날짜에 마커 추가 UI(종류 event/deadline + 라벨 입력)가 있다.
- [ ] **하드 데드라인** = 검은색 채운 칩 `⚑ 라벨`, **이벤트** = 외곽선 칩 `◆ 라벨`로 해당 날짜 칸에 표시된다(색상 막대와 구분되는 흑백).
- [ ] 마커는 DB에 저장되고 새로고침 후 유지된다.
- [ ] 마커 클릭으로 편집/삭제할 수 있다(삭제는 확인 단계).
- [ ] 사이드바에 마커 범례(⚑ 데드라인 / ◆ 이벤트)가 있다.
- [ ] Typecheck/lint 통과
- [ ] **Verify in browser:** 지정 날짜에 마커가 표시된다.

**Priority:** 4

## 6. Functional Requirements

- **FR-1:** 앱은 월이 세로로 끊김 없이 이어지는 **무한 스크롤 캘린더**를 7열로 렌더하고, 진입 시 오늘(2026-05-21)이 보이도록 스크롤해야 한다. 각 월에 sticky 헤더(월 이름+요일)를 둔다.
- **FR-2:** 사용자가 시작 칸부터 종료 칸까지 드래그하면, 시스템은 그 기간을 선택 상태로 하이라이트하고 드래그 종료 시 할일 등록 창을 열어야 한다.
- **FR-3:** 등록 창에서 제목 입력 + 프로젝트 선택 + 태스크종류 선택 후 저장하면, 시스템은 시작일~종료일 기간의 할일을 생성하고 캘린더에 막대 블럭으로 표시해야 한다.
- **FR-4:** 막대는 여러 날(기간)에 걸쳐 연속 표시되며, 주 경계를 넘으면 주별로 분할되어 연속처럼 보여야 한다.
- **FR-5:** 같은 기간에 겹치는 막대들은 자동으로 별도 레인에 세로로 쌓아 모두 보이게 해야 한다.
- **FR-6:** 시스템은 막대 배경색을 `applyTone(프로젝트.color, 태스크종류)`(dark/tint)로 계산하고, 텍스트 색을 배경 명도에 맞춰 흑/백 자동 대비 처리해야 한다.
- **FR-7:** 각 할일은 정확히 하나의 프로젝트와 하나의 태스크종류에 속해야 한다.
- **FR-8:** 태스크종류는 전역으로 공유되어 모든 프로젝트에서 동일한 톤으로 적용되어야 한다.
- **FR-9:** 시스템은 오른쪽 할일 리스트에 할일을 시작일 오름차순으로 나열해야 한다.
- **FR-10:** 사용자가 막대를 클릭하면 시스템은 편집 창(제목/기간/프로젝트/태스크종류 수정, 삭제)을 제공해야 한다.
- **FR-11:** 사용자가 막대를 다른 날짜로 드래그하면, 시스템은 기간 길이를 유지한 채 시작·종료일을 이동시켜야 한다.
- **FR-12:** 시스템은 클릭(편집)과 드래그(이동)를 임계 이동거리로 구분해야 한다.
- **FR-13:** 사용자는 프로젝트(이름/색 — 8색 팔레트)와 태스크종류(이름/톤)를 생성/수정/삭제할 수 있어야 한다.
- **FR-14:** 시스템은 상단 프로젝트 탭으로 "전체(통합)"·개별 프로젝트 뷰를 전환 제공하고, 통합 뷰에서는 색으로 프로젝트를 구분해야 한다.
- **FR-15:** 사용자는 토글로 프로젝트 단위 표시/숨김을 제어할 수 있어야 하며, 프로젝트(8색)·태스크종류(톤)·마커(⚑/◆) 범례를 제공해야 한다.
- **FR-16:** 모든 데이터(프로젝트, 태스크종류, 할일, 마커, 토글 상태)는 IndexedDB에 저장되어 새로고침·재방문 후에도 유지되어야 한다.
- **FR-17:** 모든 파괴적 작업(할일/프로젝트/태스크종류/마커 삭제)은 확인 단계를 거쳐야 한다.
- **FR-18:** 사용자는 특정 날짜에 마커(이벤트/하드 데드라인 + 라벨)를 추가할 수 있어야 하며, 시스템은 이를 흑백 기호(⚑ 데드라인 / ◆ 이벤트)로 색상 막대와 구분해 표시해야 한다.
- **FR-19:** UI(그리드·타이포·사이드바)는 모노크롬/하이라인이어야 하며, **색은 할일 막대에만** 사용해야 한다. 캘린더 그리드는 선 없는 미니멀 형태다.
- **FR-20:** 타이포는 영문 Plain(400, 상용) / 숫자·한글 IBM Plex Sans KR(무료), 자간 -3%를 적용해야 한다 (docs/design/typography.md).

## 7. Non-Goals (Out of Scope)

- 시(時):분(分) 단위 시간 일정 — **날짜 단위만** 지원
- 알림/리마인더, 반복(매주/매월) 일정
- 멀티 기기 동기화, 로그인/계정, 클라우드 백업
- 주간(week)·일간(day) 뷰 — **먼슬리 무한 스크롤만**
- 월 페이징 UI(이전/다음 버튼 중심 탐색) — **무한 스크롤로 대체**
- 캘린더 공유/협업/권한 관리
- 외부 캘린더(Google/Outlook) 가져오기·내보내기
- 태스크종류의 프로젝트별 개별 정의 — **전역 공유만** 지원
- UI 전반의 컬러 사용 — **색은 할일 막대에만**, 나머지는 모노크롬
- 모바일 터치 드래그 최적화 (데스크톱 마우스 우선; 모바일은 best-effort)
- 할일 완료 체크/진행률 (Open Question 참고)

## 8. Design Considerations

> 시각 레퍼런스(확정): [docs/design/mockups/calendar-layouts.html](../docs/design/mockups/calendar-layouts.html) **F 탭**.

- **디자인 언어:** 스위스 에디토리얼 — 모노크롬 + 하이라인. 그림자 없음, 정밀한 격자감, 넉넉한 여백.
- **그리드:** **선 없는 미니멀**(셀 테두리 없음), 큰 날짜 숫자 + 여백으로 구조. 오늘 = 검은 원형, 타 월 날짜는 흐리게.
- **무한 스크롤:** 월이 세로로 연속, sticky 월 헤더(월 이름 + 요일), 로드 시 오늘로 스크롤, 상단 타이틀이 스크롤 따라 갱신.
- **색상:** 4. 색상 시스템을 따른다. 색은 막대에만 — 프로젝트=같은 톤 8색조, 태스크종류=톤(dark/tint). 텍스트 흑/백 자동 대비.
- **레이아웃:** 상단 프로젝트 탭(전체+프로젝트, 밑줄형) + 중앙 무한스크롤 캘린더 + 우측 "할일" 리스트(카운트 뱃지·날짜순·범례·＋할일추가). 좁은 화면에선 우측 리스트 접힘.
- **타이포:** 영문 Plain 400 / 숫자·한글 IBM Plex Sans KR(무료), 자간 -3%, 날짜는 tabular-nums (docs/design/typography.md).
- **마커:** 하드 데드라인 = 검은 채운 칩 `⚑`, 이벤트 = 외곽선 칩 `◆` — 색 막대와 구분.
- **상호작용 피드백:** 드래그 선택/이동 시 미리보기, 저장 시 부드러운 등장 애니메이션(과하지 않게).
- **빈 상태:** 할일이 없을 때 "일칸을 드래그해 첫 할일을 추가하세요" 안내.

## 9. Technical Considerations

- **색 합성 유틸(`lib/color/`)**: `applyTone(hex, {mode,k})` + `barText(hex)→흑/백` 두 순수 함수로 분리해 단위 테스트(color-system.md 매트릭스와 일치).
- **무한 스크롤:** 연속 주 생성 + 월 그룹(주의 목요일 기준) + sticky 월 헤더 + 오늘로 스크롤. 범위가 커지면 가상화/지연 로딩 검토. 막대의 left%/width%는 7열 기준 유지.
- **막대 레이아웃(레인) 알고리즘**: 주 단위로 막대를 끊고, 겹침 구간을 인터벌 스케줄링으로 레인에 배치한다. 순수 함수로 분리(`lib/calendar/layout.ts`)해 단위 테스트.
- **날짜 처리:** 타임존 이슈를 피하기 위해 날짜는 `YYYY-MM-DD` 문자열로 저장하고, 비교/연산은 UTC/로컬 자정 기준 유틸로 처리한다.
- **드래그 구현:** Pointer Events 기반 직접 구현하거나, 복잡도가 커지면 `dnd-kit` 도입 검토.
- **상태 관리:** Dexie `liveQuery` + React 훅으로 DB ↔ UI 반응형 동기화.
- **시드 데이터:** 기본 프로젝트 1 + 기본 태스크종류 4개(마감/작업/회의/리서치 톤) 첫 실행 주입.
- **타이포 구현:** `@font-face` + `unicode-range`로 영문=Plain400 / 숫자·한글=IBM Plex Sans KR 분리 (typography.md). IBM Plex는 Google Fonts(무료), 영문 Plain만 woff2 호스팅 필요.
- **클릭 vs 드래그 구분:** 포인터 이동 임계값(예: 4px) 기준.

## 10. Success Metrics

- 할일 1건 등록까지 **드래그 1회 + 텍스트 1줄 + (프로젝트·종류 기본값) 저장 클릭 1회** 안에 완료된다.
- 통합 뷰에서 프로젝트 3개가 **색만으로** 구분되고, 같은 프로젝트 내 태스크종류 3개가 **톤만으로** 구분된다.
- 모든 톤 단계의 막대에서 텍스트가 읽힌다(가독성 통과).
- 겹치는 일정 5건이 한 주에 있어도 모두 가려지지 않고 보인다.
- 막대 드래그 이동 후 새로고침해도 데이터가 100% 유지된다.

## 12. 디자인 스킬 활용 계획 (Design Skill Plan)

구현 워크플로우 단계별로 사용할 디자인 스킬과 시점을 정리한다. 중복되는 UI 빌드 계열은 메인 1개 + 대안으로 묶었다.

### 단계별 매핑

| 단계 | 스킬 | 목적 | 시점 (연관 US) |
|---|---|---|---|
| 0. 방향 탐색 *(선택)* | `design-plugin` | 3분할 레이아웃(사이드바·캘린더·패널) 시안 여러 개 생성·비교 후 방향 확정 | 착수 전, US-001 이전 |

| 4. 리뷰 | `vercel-agent-skills` | Web Interface Guidelines / 접근성(WCAG) 준수 점검 | 각 UI 스토리 완료 후 + 최종 |

### 권장 순서

```
brand-color-palette (색조/톤 확정)
  → ui-design-system (토큰)
  → [US-001 스캐폴딩]
  → ux-design-principles (플로우 UX)
  → ui-ux-pro-max (컴포넌트 빌드, US-003~014)
  → microinteraction (드래그/막대 모션)
  → vercel-agent-skills (UI/접근성 리뷰)
```

### 메모

- **메인 빌드 스킬 대안:** `ui-ux-pro-max` 대신 `impeccable`(craft) / `frontend-design` / `taste-skill` / `ui-design-brain` 중 택1 가능. 하나만 메인으로 쓰고 혼용하지 않는다(스타일 일관성).
- **`brand-color-palette` 산출물 → 코드 연결:** 생성된 색조/톤 값이 US-002의 시드 데이터(`Project.hue`, `TaskType.{saturation,lightness}`)와 US-006의 색 합성 유틸 입력값이 된다.
- **Figma는 비범위:** 코드 우선 빌드라 Figma MCP는 사용하지 않음(별도 디자인 핸드오프가 필요할 때만).
- 색상은 이 앱의 차별점이므로 `brand-color-palette` → `microinteraction` 두 스킬에 특히 공을 들인다.

## 13. Ralph 자동 구현 운영 계획 (Hybrid)

전체 과정을 Ralph로 자동화하되, **사람 감각이 필요한 1회성 결정은 앞에서 확정**하고 **반복 구현만 Ralph**에 맡기는 하이브리드로 진행한다.

### 단계

```
[Phase A — 사람과 함께 (완료)]
  ✅ brand-color-palette → 색조 8 + 톤 4, 32조합 WCAG AA↑ (docs/design/color-system.md)
  ✅ prd.json 생성 (scripts/ralph/prd.json, 16 스토리 passes:false)
  ✅ 디자인 지침 내장 CLAUDE.md (scripts/ralph/CLAUDE.md)
  ◻ (선택) ui-design-system 토큰, design-plugin 레이아웃 시안

[Phase B — Ralph 자동 (백그라운드)]
  scripts/ralph/ralph.sh 루프: 우선순위순 1스토리/반복 → 타입체크·린트·테스트 → 커밋 → passes:true

[Phase C — 사람 패스 (Ralph 후)]
  "Verify in browser" 항목 시각 검증 + microinteraction 모션 폴리시 + vercel-agent-skills 접근성 리뷰
```

### Ralph 안에서의 스킬 사용 (제약 반영)

- 헤드리스(`claude --print`) 반복이라 **디자인 지침을 `scripts/ralph/CLAUDE.md`에 내장**해 스킬 미로딩에도 일관성 유지. 스킬이 로딩되면 해당 스토리에서 `ui-ux-pro-max`/`microinteraction`/`ux-design-principles` 호출 허용.
- **색은 절대 재생성 금지** — `docs/design/color-system.md` 확정값만 사용.
- **시각 검증 불가** — Ralph는 타입체크/린트/테스트까지만, "Verify in browser"는 `progress.txt`에 남겨 사람이 Phase C에서 확인.


## 14. v2 기능 (US-018~025)

US-001~017(v1)은 구현·배포 완료. 아래는 추가 요청 기능. 확정 결정: **공유=스냅샷 발행(Vercel Blob, 로컬 우선 유지)**, **태스크종류=프로젝트별(새 프로젝트에 기본4종 복제 시드 + 기존 전역 복제 마이그레이션)**, **히트맵=모노크롬 농담**. "프로젝트별로 쪼개기"(US-020/021)는 v1의 전역 공유 모델을 대체한다.

### 단계
1. US-018·019 (소규모·독립) → 2. US-020·021 (프로젝트별 데이터 모델, 함께) → 3. US-022 (히트맵) → 4. US-023·024·025 (외부 공유, 백엔드)

### US-018: 프로젝트 순서 수동 변경
**Description:** 사용자로서 프로젝트 순서를 드래그로 바꿔 원하는 순서로 정렬한다.
**Acceptance Criteria:**
- [ ] 상단 프로젝트 탭(또는 사이드 범례)에서 드래그로 순서 변경
- [ ] 변경된 `order`가 저장되고 새로고침·재방문 후 유지
- [ ] 탭/범례/뷰 전반에 새 순서 반영
- [ ] 클릭(뷰 전환)과 드래그(정렬) 구분
- [ ] Typecheck/lint, 단위 테스트(정렬 로직), Verify in browser
**Priority:** 8

### US-019: 일정별 노트
**Description:** 사용자로서 각 일정에 메모(노트)를 추가해 세부 내용을 적는다.
**Acceptance Criteria:**

- [ ] 편집 팝오버에 노트 textarea, 저장/수정
- [ ] 노트 있는 일정은 막대/리스트에 표시(작은 점·아이콘)
- [ ] 노트가 인쇄·공유 스냅샷에 포함
- [ ] 새로고침 후 유지, Typecheck/lint, Verify in browser
**Priority:** 8

### US-020: 태스크 종류 프로젝트별 전환
**Description:** 개발자/사용자로서 태스크 종류를 프로젝트별로 따로 관리한다(전역 공유 폐지).
**Acceptance Criteria:**
- [ ] `TaskType.projectId` 추가 — 각 종류는 한 프로젝트에 소속
- [ ] 새 프로젝트 생성 시 기본 4종(리서치/회의/작업/마감, color-system 톤) 자동 시드
- [ ] **마이그레이션**: 기존 전역 종류를 각 프로젝트로 복제하고 일정의 `taskTypeId`를 같은 프로젝트의 종류로 재연결 (Dexie 버전 업그레이드)
- [ ] 종류 관리·범례·생성·필터가 현재 프로젝트의 종류만 대상으로 동작
- [ ] 톤 중복 방지·기본종류 보호는 프로젝트 단위로 유지
- [ ] Typecheck/lint, 마이그레이션·CRUD 테스트, Verify in browser
**Priority:** 9

### US-021: 데드라인·이벤트(마커) 프로젝트별 전환
**Description:** 사용자로서 마커(이벤트/하드 데드라인)도 프로젝트별로 관리한다.
