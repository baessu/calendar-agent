# 색상 시스템 — 이중축 (Dual-Axis Color System)

> 생성: `brand-color-palette` 스킬 기반 (HSB/Hue 원리 + WCAG 검증)
> 막대색 공식: **`hsl(프로젝트.색조, 태스크종류.채도S, 태스크종류.명도L)`**
> 텍스트색: 배경과 흰색/검정의 WCAG 대비를 각각 계산해 **높은 쪽 자동 선택**

```ts
// lib/color/compose.ts
/** 막대 배경색 = 프로젝트 색조 + 태스크종류 톤 */
export function barBg(hue: number, s: number, l: number): string {
  return `hsl(${hue} ${s}% ${l}%)`;
}

/** WCAG 대비가 높은 쪽(흰/검정)을 자동 선택 */
export function barText(hue: number, s: number, l: number): "#FFFFFF" | "#1A1A1A" {
  const lum = hslLuminance(hue, s, l);
  const cWhite = (1.0 + 0.05) / (lum + 0.05);
  const cBlack = (lum + 0.05) / (0.0 + 0.05);
  return cWhite >= cBlack ? "#FFFFFF" : "#1A1A1A";
}
// hslLuminance: HSL→sRGB→상대휘도(0.2126R+0.7152G+0.0722B, 선형화 후)
```

## 5. 메모 / 향후

- **퍼셉추얼 균일성:** HSL은 같은 L이라도 색조마다 체감 밝기가 달라(특히 앰버), 톤 단계가 미세하게 들쭉날쭉할 수 있다. 더 깔끔히 가려면 **OKLCH**로 전환(같은 L = 같은 체감 밝기) 권장 — v1은 HSL + 조합별 WCAG 검증으로 충분.
- **텍스트 자동색은 조합별 계산이 정답** (위 매트릭스가 근거). 단순 L 임계값(예: L≥55 검정)으로 근사해도 대부분 일치하나, 경계 색조(잉크블루 T2 등)에서 어긋날 수 있으니 `barText()`로 실제 대비 계산 권장.
- 톤 단계를 늘리면(>4) 중간 명도 구간에서 텍스트 대비가 떨어지니, 4단계 유지 권장.
